5.2 Coding Standards & Best Practices

Modular Code & Reusability

All scripts are organized into distinct logical functions with clear input and output parameters. Global configuration constants (such as random seeds, paths, and thresholds) are defined at the top of each file.

Naming Conventions

The code strictly adheres to Python's standard style guides:

  • Variables and Functions: Written in snake_case (e.g., calculate_risk_score, data_path).
  • Classes: Written in PascalCase (e.g., ModelPipeline).
  • Constants: Written in UPPER_SNAKE_CASE (e.g., RANDOM_STATE = 42, YOUDEN_THRESHOLD = 0.4922).
  • SQL Schemas: Follows star-schema naming conventions with fact_ and dim_ prefixes.

Security & Error Handling

  • No Sensitive Credentials: No API keys or credentials are stored in clear text.
  • Graceful Degradation: The web app uses try-except blocks when computing SHAP values or loading pickle models. If SHAP computation fails due to missing dependencies, the app gracefully falls back to showing prediction scores and warnings without crashing.
  • Type Castings: Input inputs from form elements are cast and verified (e.g., casting age midpoints to numeric values).

5.3 Database Schema Mapping (SQL DDL)

Below is a snippet of the DDL schema utilized to configure the Star Schema tables in SQL Server:

-- 1. Create Geography Dimension
CREATE TABLE dim_Geography (
    State NVARCHAR(100) PRIMARY KEY,
    Region NVARCHAR(50),
    Population_M FLOAT,
    State_HD_Prevalence FLOAT,
    Obesity_Rate_Pct FLOAT,
    Smoking_Rate_Pct FLOAT,
    Uninsured_Rate_Pct FLOAT,
    Median_Income_K FLOAT,
    State_Health_Rank INT
);

-- 2. Create Patient Dimension
CREATE TABLE dim_Patients (
    PatientID VARCHAR(50) PRIMARY KEY,
    Sex INT,
    AgeCategory NVARCHAR(50),
    Age_Numeric INT,
    Age_Group NVARCHAR(50),
    Race NVARCHAR(100)
);

-- 3. Create Fact Table
CREATE TABLE fact_Diagnosis (
    PatientID VARCHAR(50) PRIMARY KEY,
    State NVARCHAR(100),
    DateID INT,
    HeartDisease INT,
    Risk_Score FLOAT,
    Risk_Tier NVARCHAR(50),
    ML_Probability FLOAT,
    ML_Prediction INT,
    Patient_Segment INT,
    Patient_Segment_Label NVARCHAR(100),
    FOREIGN KEY (PatientID) REFERENCES dim_Patients(PatientID),
    FOREIGN KEY (State) REFERENCES dim_Geography(State)
);

5.4 Version Control & Collaboration Strategy

The project is hosted on GitHub to manage version control and collaborative development. Our workflow follows a structured Git model:

  • Branching Strategy: A clean main branch represents production-ready code. Features were developed in local branches and merged via documented commits.
  • Commit Messages: Follow clear descriptions (e.g., "Add Feature Engineering notebook, team credits, and HTML documentation for PDF").
  • Large File Handling: Git LFS (Large File Storage) or `.gitignore` was configured for files exceeding GitHub's 100MB limit (like raw `.xlsx` datasets and `.pkl` models) to prevent repository bloating.

5.5 Execution Guide

System Requirements

  • Operating System: Windows 10/11 or Linux
  • Python Version: 3.8, 3.9, or 3.10
  • RAM: Minimum 8 GB (16 GB recommended for running full Optuna pipelines)
  • Disk Space: 500 MB (excluding raw data files)

Step 1: Clone the Repository

git clone https://github.com/nhahub/NHA-4-210.git
cd NHA-4-210

Step 2: Install Python Dependencies

pip install -r "Machine Learning/requirements.txt"

Step 3: Launching the Web App

To run the Streamlit real-time risk diagnostic application, execute the following in your shell:

streamlit run "Machine Learning/app.py"

This will start a local server and open your default browser to http://localhost:8501.

Step 4: Executing the ML Pipelines

# Run V1 Baseline Training (saves to outputs/)
python "Machine Learning/pipelines/ml_pipeline.py"

# Run V2 Optuna Tuning and Training (saves to outputs/v2/)
python "Machine Learning/pipelines/ml_pipeline_v2.py"
